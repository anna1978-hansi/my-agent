🌟 1. 突破流式渲染性能瓶颈（最高含金量）
普通写法： 支持解析 AI 输出的 Markdown 并且渲染在页面上。
高级写法：
针对 AI Agent 流式输出场景（高频 Token 接收），重构了 Markdown 渲染链路。通过防抖调度（Throttling/Debounce）与增量 AST 解析策略，避免了超大文档在每个 token 到达时触发的全量重绘（Reflow/Repaint），保障了长列表与长代码块在解析时的页面帧率（FPS）稳定在 60，彻底解决输入卡顿问题。
🌟 2. 基于 AST 的异步语法高亮引擎 (Shiki)
普通写法： 引入 Shiki 库实现了代码块语法高亮。
高级写法：
摒弃传统基于正则的简易解析，主导搭建了基于抽象语法树（AST）的 Markdown 引擎（如 remark/rehype 生态）。优雅解决了 Shiki 高亮引擎的异步初始化与同步渲染生命周期冲突问题。实现 Shiki 实例的单例缓存与按需加载，大幅降低了内存开销与首次渲染的白屏时间。
🌟 3. 自定义渲染器（Custom Renderer）与 Diff 视图
普通写法： 给代码块加了复制按钮和折叠功能，支持了旧代码和新代码的对比。
高级写法：
采用自定义渲染器（Custom Renderer）模式将纯文本节点转化为高阶 UI 组件。实现了高度定制化的代码块生态：包含一键复制、语言嗅探、行号映射以及超长代码段的智能折叠；针对 Agent 修改代码场景，通过 AST 拦截自定义了 Diff 视图（Merge 场景），精准实现了增删代码行（Old/New）的直观视觉差异（Visual Diff）展现。

# Agent Markdown 阅读器工程化开发任务书

## 1. 背景与目标
我们的 AI Agent 平台会输出包含大量代码、系统指令的 Markdown 流。之前的纯文本渲染过于简陋，现在需要将其升级为“企业级阅读器”。必须解决代码块的复杂 UI 交互、Diff 对比渲染以及流式输出时的渲染性能衰减问题。

## 2. 全局强制约束（AI 必须遵守）
1. **测试驱动：** 每个 Task 完成后，**必须**提供一个独立的测试/模拟脚本（Node.js script 打印 AST 或 HTML 结构，或模拟前端渲染输出）。
2. **框架无关或指明框架：** 核心解析逻辑需与 UI 框架解耦。如果涉及具体 UI 组件（如折叠、复制按钮），请提供清晰的数据结构或 React/Vue 组件范例。
3. **运行指南：** 在每次你输出代码的最后，**必须用明显的区块（如 `### 如何运行测试`）告诉我如何在本地跑起这个测试文件**。

## 3. 架构设计方向
- 弃用简单的 `marked.js` 默认行为，采用基于 AST（抽象语法树）的解析方案（推荐使用 `unified` / `remark` / `rehype` 生态，或者重写 `marked` 的 renderer）。
- **Shiki 集成：** 使用 `shiki` 替代 `prism`。由于 `shiki` 是异步的，需要设计缓存和初始化逻辑。
- **自定义组件映射：** 将拦截到的 `<pre><code>` 节点转化为包含 `language`、`raw_code`、`diff_lines` 的数据对象，以便 UI 层渲染。

## 4. 任务优先级与拆解 Plan

我们将按照从核心解析到高阶 UI 的顺序执行。**请等待我给出 "执行 Task X" 的指令后再输出代码。**

### 📦 Task 1: 核心引擎与 Shiki 异步高亮封装
- **目标：** 搭建 Markdown 到 HTML/AST 的解析管线，并接入 Shiki 进行代码高亮。
- **要求：**
  - 实现一个 `MarkdownEngine` 类。
  - 内部实现 Shiki 的单例模式（保证 `getHighlighter` 只初始化一次，并缓存常用主题和语言）。
  - 支持解析普通 Markdown 并将代码块输出为带有 Shiki CSS token 的安全 HTML。
  - **测试要求：** 编写测试脚本，输入一段包含 JS 代码块的 MD 字符串，异步输出经 Shiki 处理后的完整 HTML 片段。

### 📦 Task 2: 代码块高阶 UI 数据提取 (行号、语言、复制)
- **目标：** 不仅要高亮，还需要为前端 UI 框架提供渲染“复制按钮、代码折叠”所需的数据。
- **要求：**
  - 拦截代码块渲染（Custom Renderer）。
  - 提取出元数据：`language`（语言标识）、`rawCode`（用于复制的纯净文本）、`lineCount`（用于计算是否需要展示“展开/折叠”按钮的行号）。
  - 针对超过一定行数（例如 15 行）的代码块，在返回的结构中标记 `isCollapsible: true`。
  - **测试要求：** 输入一段超过 20 行带有语言标识的代码块，验证输出的元数据对象是否准确包含了 `rawCode` 和 `isCollapsible` 标志。

### 📦 Task 3: MERGE 场景下 Diff 视图解析优化
- **目标：** 优化 AI 输出修改代码时的旧/新代码对比效果（类似 Github 提交记录）。
- **要求：**
  - 针对语言声明为 `diff` 或包含 `+` `-` 前缀的代码块进行特殊处理。
  - 按行解析：将以 `-` 开头的行标记为 `deletion`（红色背景类名），以 `+` 开头的行标记为 `addition`（绿色背景类名）。
  - 清理多余的前缀符号，确保通过复制按钮复制的代码不带无用的 `+` `-`。
  - **测试要求：** 输入一段 `diff` 格式的 Markdown，打印出解析后的结构，证明增加/删除的行被正确赋予了特殊的类名，且生成的 `rawCode` 是干净的代码。

### 📦 Task 4: 流式渲染性能优化 (Throttling & 边界处理)
- **目标：** 防止 SSE 流式不断吐出 token 时，每次都全量解析重绘导致浏览器卡死。
- **要求：**
  - 实现一个带防抖/节流的渲染调度包装器（`StreamRenderer`）。
  - 当不断调用 `appendToken(text)` 时，限制解析器的执行频率（例如 `requestAnimationFrame` 结合最高 50ms 一次触发）。
  - 边界处理：处理代码块在闭合之前（只有 ` ``` ` 没有尾部 ` ``` `）的残缺 AST 容错渲染，防止渲染崩溃或高亮乱码。
  - **测试要求：** 模拟一个高频的 `setInterval` 每 5ms 输入一个字符，验证 `render` 回调在 1 秒内只被触发有限次（如 20 次以内），并且残缺的代码块也能正常 fallback 为普通文本或默认高亮。
